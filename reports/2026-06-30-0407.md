### A1：openclaw/openclaw #81484 — Discord 群组回复回归：消息负载损坏和重复发送循环

**链接：** https://github.com/openclaw/openclaw/issues/81484
**评分：** 17/25（深度:4，社区关注:3，范围清晰度:4，分析价值:3，维护者参与:3）

#### 背景

OpenClaw 2026.5.7 版本中，Discord 私信功能正常，但群组/服务器频道回复间歇性失败。外发消息缺少必需的 `message` 负载字段，导致畸形请求。在某些情况下，系统还会进入重复发送循环——同一条回复被反复发送。

#### 社区评估

- **活跃度：** 👍2，5 条评论
- **已有实现：** 无
- **维护者态度：** 无人类维护者，Codex bot "保持开放"
- **判断：** 可行。清晰的回归，影响 Discord 群组用户的核心体验

#### 问题分析

Discord 私信路径和群组频道路径使用了不同的消息构造逻辑。群组路径在构造外发消息时，`message` 负载字段可能未正确填充（空字符串或缺失的 content 对象）。Discord API 对群组消息有更严格的 schema 验证，畸形请求会被拒绝。重复发送循环则表明重试逻辑未能正确识别外发失败——可能将 API 层的 schema 错误误判为临时网络问题，不断重试。

#### 修复方案

**根因：** Discord 群组频道的消息构造逻辑与私信路径不同，`send_message` 的 payload 构造可能缺少必需的 `content` 或 `embeds` 字段。

**修改思路：**
1. 对比私信和群组频道的消息构造代码路径差异
2. 在消息发送前添加 payload schema 断言——确保必需的 `message` 字段不为空
3. 修复重试逻辑：Discord API 的 schema 错误（400）不应触发重试

**涉及文件：** Discord 频道插件，消息构造和发送模块

**风险评估：** 低。修复范围限定在 Discord 消息构造层。

### A3：earendil-works/pi #6108 — Release 二进制文件在 /reload 时重复执行扩展依赖的副作用

**链接：** https://github.com/earendil-works/pi/issues/6108
**评分：** 15/25（深度:4，社区关注:2，范围清晰度:4，分析价值:3，维护者参与:2）

#### 背景

pi 的 Linux release 二进制文件在执行 `/reload` 重载扩展时，会重新评估扩展的依赖模块。这导致依赖模块的副作用（如注册主题、全局状态初始化）被重复执行。已知可复现案例：`@plannotator/pi-extension` 依赖 `@pierre/diffs`，后者在模块加载时注册内置主题——每次 `/reload` 都会重复注册。

#### 社区评估

- **活跃度：** 0👍，2 条评论
- **已有实现：** 无
- **维护者态度：** 无回复
- **判断：** 可行。有趣的技术问题——模块副作用和热重载的交互

#### 问题分析

Node.js 的模块缓存（`require.cache`）通常在首次 `require` 后不会重新执行模块代码。但 pi 的 `/reload` 机制可能绕过了这个缓存——清除了 `require.cache` 中的条目以允许扩展更新，但这也导致依赖模块被重新执行。问题的核心是：如何区分"需要重新加载的扩展代码"和"不应重复执行的依赖副作用"。

#### 修复方案

**修改思路：**
1. 在 `/reload` 时只清除扩展入口文件的缓存，保留依赖模块的缓存
2. 或者：要求依赖模块将副作用设计为幂等的（如主题注册前检查是否已存在）
3. 长期方案：扩展系统的模块加载器应支持"标记为 side-effect-free 的模块不重新加载"

**涉及文件：** 扩展加载器/重载模块

**风险评估：** 低。修改重载逻辑，可向后兼容。
