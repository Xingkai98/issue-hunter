#### Problem

This suggests the event type enum or handler registration is missing these two event types in the Slack Socket Mode adapter, while the HTTP (Events API) path may handle them correctly.

**Root cause:** The Slack Socket Mode event dispatcher likely has an incomplete event type mapping. Check if `reaction_added`/`reaction_removed` are registered in the Socket Mode handler's event routing table.

**Proposed approach:**
1. Compare the event type registry between HTTP Events API path and Socket Mode path

**Files to modify:**

**Risk:** Low. Adding missing event types is mechanical. The main risk is that Socket Mode may have architectural constraints preventing reaction event delivery (different subscription model), which would require deeper investigation.

### A2: openclaw/openclaw #56217 — Secret provider crash-loop exhausts 1Password rate limits

**Link:** https://github.com/openclaw/openclaw/issues/56217
**Score:** 16/25 (depth: 4, engagement: 2, interest: 3, clarity: 4, value: 3)

#### Background

OpenClaw uses external secret providers (1Password, etc.) to resolve API keys and credentials. When 1Password service account credentials fail (rate limit, network issue, token expiry), the gateway enters a crash-loop via launchd `KeepAlive`, repeatedly invoking `op read` for every configured secret provider on each restart. This quickly exhausts 1Password's account-wide rate limits.

#### Community Assessment

- **Last meaningful activity:** 2026-06-27
- **Existing implementation:** None
- **Maintainer stance:** No human maintainer engagement yet
- **Verdict:** VIABLE. Operational reliability issue with concrete failure mode

#### Problem

The crash-loop is caused by two interacting issues:
1. launchd `KeepAlive` restarts the gateway unconditionally on exit
2. On startup, all secret providers are resolved eagerly — before any health checks

When the 1Password API returns 429 (rate limited), the gateway fails to start, launchd restarts it, and the cycle repeats — each iteration making more `op read` calls and worsening the rate limit situation.

#### Fix Strategy

**Proposed approach:**
1. Add exponential backoff with jitter to secret provider resolution on startup
2. Implement a circuit breaker — after N consecutive failures, stop retrying and signal the operator
3. Add a startup health gate — don't resolve all providers eagerly; lazy-resolve on first use
4. Optionally: detect 429 responses specifically and pause before retrying

**Files to modify:**
- Secret provider resolution module (`src/infra/secrets/` or similar)
- Gateway startup/bootstrap sequence
- launchd plist configuration (add `ThrottleInterval`)

**Risk:** Medium. Changing startup behavior could delay gateway readiness. Needs to be gated behind configuration.

1. Extensions receiving `agent_end` events need to mutate the session tree, but it's not safe during event dispatch

### B1: anthropics/claude-agent-sdk-python #1060 — PreToolUse "defer" does not end the query

**Link:** https://github.com/anthropics/claude-agent-sdk-python/issues/1060
**Score:** 18/25 (depth: 5, engagement: 2, interest: 5, clarity: 4, value: 2)

#### Background

The official Anthropic Claude Agent SDK for Python (0.2.x). When a `PreToolUse` hook returns `permissionDecision: "defer"` for an MCP tool call, the SDK correctly stops the tool from executing and emits `ResultMessage.stop_reason == "tool_deferred"`. However, before that ResultMessage is emitted, the subprocess sometimes gives the model one additional opportunity to respond — causing inconsistent behavior where the model either gets or doesn't get a "wrap-up" turn after deferral.

#### Community Assessment

- **Last meaningful activity:** Recent — user ran 6× repro showing 3/5 runs had `wrapup_turn=True` (inconsistent)
- **Existing implementation:** None
- **Maintainer stance:** No Anthropic response yet
- **Verdict:** HIGH VALUE. Official Anthropic SDK bug with quantitative repro data. Great analysis opportunity.

#### Problem

From the repro data: 6 runs of the same defer scenario produced:
- Run 1: `wrapup_turn=False`
- Run 2: `wrapup_turn=True`
- Run 3: `wrapup_turn=True`
- Run 4: `wrapup_turn=True`
- Run 5: `wrapup_turn=False`
- Run 6: (implied inconsistent)

The defer decision should be terminal — no further model turns after defer. The inconsistency suggests a race condition or state leakage between the subprocess iterator and the conversation boundary detection.

#### Fix Strategy

**Root cause:** The subprocess iterator likely doesn't properly signal "terminal" after a defer decision. The SDK may be checking a condition that is sometimes true (model has more to say) and sometimes false (model exhausted), when defer should always short-circuit.

**Proposed approach:**
1. In the PreToolUse hook handler, when defer is returned, set a terminal flag on the conversation state
2. Check this flag before giving the model another response opportunity
3. The regression test suggested by the reporter: assert exactly one deferred tool receipt is emitted per defer decision

**Files to modify:**
- SDK core — likely `src/claude_agent_sdk/` in the hook dispatch or conversation loop
- Regression test file

**Risk:** Low. The fix is adding a guard condition. The main risk is ensuring the terminal flag doesn't leak across conversation boundaries.

### B3: headroomlabs-ai/headroom #1495 — Proxy retry: upstream 529 overloaded treated differently from 429

**Link:** https://github.com/headroomlabs-ai/headroom/issues/1495
**Score:** 14/25 (depth: 3, engagement: 1, interest: 3, clarity: 3, value: 4)

#### Background

headroom (52k stars, created Jan 2026) is a fast-growing token compression library. Its proxy layer forwards requests to upstream LLM providers. When upstream returns HTTP 529 (overloaded), the proxy does not apply the same retry logic as for HTTP 429 (rate limited), causing unnecessary failures.

#### Community Assessment

- **Last meaningful activity:** 2026-06-27
- **Existing implementation:** PR title format suggests this may already be a fix — needs verification
- **Maintainer stance:** Unknown
- **Verdict:** MODERATE. Fast-growing project worth watching. Issue may already have a fix.

#### Problem

HTTP 529 (Service Overloaded) is semantically similar to 429 (Too Many Requests) — both indicate the upstream is temporarily unable to handle the request. The proxy's retry logic only handles 429, so 529 responses are passed through as failures to the client instead of being retried.

#### Fix Strategy

**Root cause:** The retry condition in the proxy forwarder checks for status code 429 but not 529.

**Proposed approach:** Add 529 to the retry-eligible status codes list alongside 429. Apply same exponential backoff and jitter.

**Files to modify:** Proxy forwarder module — retry condition check

**Risk:** Trivial. One-line change adding a status code to a condition.
