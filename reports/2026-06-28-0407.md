### A1：openclaw/openclaw #56653 — Slack 回应事件在 Socket Mode 下无法送达

**链接：** https://github.com/openclaw/openclaw/issues/56653
**评分：** 17/25（深度:4，维护者参与:2，社区关注:4，范围清晰度:4，分析价值:3）

#### 背景

OpenClaw（38 万星）是多平台 AI agent 运行时。其 Slack 集成支持 Socket Mode 用于多账户部署。尽管 OAuth 权限、事件订阅和配置在 5 个 bot 账户上全部正确，`reaction_added` 和 `reaction_removed` 事件始终无法送达网关——即使在 trace 级别日志中也看不到任何事件。

#### 社区评估

- **最近有意义的活动：** 2026-06-27（Codex 审查），一名社区用户确认相同问题
- **已有实现：** 无
- **维护者态度：** Codex bot 标记"保持开放"，无人类维护者回复
- **判断：** 可行。2 个用户确认，清晰的集成 bug，有具体的复现环境

#### 问题分析

Slack 的 Socket Mode 通过 WebSocket 传递事件。问题仅局限于 `reaction_added`/`reaction_removed` 两种事件——其他 Slack 事件均正常送达。这表明 Socket Mode 适配器中的事件类型枚举或处理器注册缺少了这两种事件类型，而 HTTP（Events API）路径可能处理正确。

#### 修复方案

**根因：** Slack Socket Mode 事件分发器的事件类型映射表不完整。需检查 `reaction_added`/`reaction_removed` 是否已在 Socket Mode 处理器的事件路由表中注册。

**修改思路：**
1. 对比 HTTP Events API 路径和 Socket Mode 路径的事件类型注册表
2. 在 Socket Mode 事件路由表中添加 `reaction_added` 和 `reaction_removed`
3. 添加集成测试验证 Socket Mode 能正常传递这两种事件

**涉及文件：** Slack 频道插件的 Socket Mode 处理器（可能在 `extensions/slack/src/socket-mode/`），事件类型注册/常量文件

**风险评估：** 低。添加缺失的事件类型是机械性操作。主要风险是 Socket Mode 可能存在架构限制导致无法传递回应事件（不同的订阅模型），这需要更深入调查。

### A2：openclaw/openclaw #56217 — 密钥提供方崩溃循环耗尽 1Password 速率限制

**链接：** https://github.com/openclaw/openclaw/issues/56217
**评分：** 16/25（深度:4，维护者参与:2，社区关注:3，范围清晰度:4，分析价值:3）

#### 背景

OpenClaw 使用外部密钥提供方（1Password 等）来解析 API key 和凭证。当 1Password 服务账户凭证解析失败（速率限制、网络问题、token 过期）时，网关通过 launchd `KeepAlive` 进入崩溃循环，每次重启都为每个已配置的密钥提供方重复调用 `op read`。这会迅速耗尽 1Password 的账户级速率限制。

#### 社区评估

- **最近有意义的活动：** 2026-06-27
- **已有实现：** 无
- **维护者态度：** 无人类维护者参与
- **判断：** 可行。运维可靠性问题，有具体的故障模式描述

#### 问题分析

崩溃循环由两个交互因素导致：
1. launchd `KeepAlive` 在网关退出时无条件重启
2. 启动时所有密钥提供方被急切解析——在任何健康检查之前

当 1Password API 返回 429（被限速）时，网关启动失败，launchd 重启它，循环往复——每次迭代都会发起更多 `op read` 调用，进一步恶化速率限制状况。

#### 修复方案

**修改思路：**
1. 在启动时为密钥提供方解析添加带抖动的指数退避
2. 实现熔断器——连续 N 次失败后停止重试并通知运维人员
3. 添加启动健康门控——不要急切解析所有提供方，改为首次使用时懒加载
4. 可选：专门检测 429 响应，在重试前暂停

**涉及文件：** 密钥提供方解析模块（`src/infra/secrets/` 或类似路径），网关启动/引导序列，launchd plist 配置（添加 `ThrottleInterval`）

**风险评估：** 中等。修改启动行为可能延迟网关就绪。需要通过配置开关控制。

3. 实现为在每次 agent 轮次结束时排空的空闲队列

**涉及文件：** Agent 循环核心——添加空闲回调队列，扩展 API 接口——暴露 `runWhenIdle`，已损坏的示例 `examples/extensions/reload-runtime.ts`——更新使用新 API

### B1：anthropics/claude-agent-sdk-python #1060 — PreToolUse "defer" 未正确终止查询

**链接：** https://github.com/anthropics/claude-agent-sdk-python/issues/1060
**评分：** 18/25（深度:5，维护者参与:2，社区关注:5，范围清晰度:4，分析价值:2）

#### 背景

Anthropic 官方 Claude Agent SDK for Python（0.2.x）。当 `PreToolUse` 钩子对 MCP 工具调用返回 `permissionDecision: "defer"` 时，SDK 正确地阻止了工具执行并发出 `ResultMessage.stop_reason == "tool_deferred"`。然而，在该 ResultMessage 发出之前，子进程有时会额外给模型一次响应机会——导致行为不一致：defer 之后模型随机获得或不获得"收尾"轮次。

#### 社区评估

- **最近有意义的活动：** 近期——用户运行了 6 次复现，其中 3/5 次出现 `wrapup_turn=True`（不一致）
- **已有实现：** 无
- **维护者态度：** Anthropic 尚未回复
- **判断：** 高价值。Anthropic 官方 SDK 的 bug，有定量复现数据。绝佳的分析机会。

#### 问题分析

复现数据：同一 defer 场景的 6 次运行产生不一致结果——3 次有 wrapup_turn，2 次没有。defer 决定应该是终态的——defer 之后不应再有模型轮次。不一致性表明子进程迭代器和对话边界检测之间存在竞态条件或状态泄漏。

#### 修复方案

**根因：** 子进程迭代器在 defer 决定后可能未正确发出"终止"信号。SDK 可能在检查某个条件时（模型还有话要说 vs 模型已穷尽），而 defer 应始终短路此检查。

**修改思路：**
1. 在 PreToolUse 钩子处理器中，当返回 defer 时，在对话状态上设置终止标志
2. 在给模型额外响应机会之前检查此标志
3. 用户建议的回归测试：断言每次 defer 决定只发出一个延迟工具收据

**涉及文件：** SDK 核心——钩子派发或对话循环（可能在 `src/claude_agent_sdk/`），回归测试文件

**风险评估：** 低。修复是添加守卫条件。主要风险是确保终止标志不会跨对话边界泄漏。

### B3：headroomlabs-ai/headroom #1495 — 代理重试：上游 529 过载应与 429 同等处理

**链接：** https://github.com/headroomlabs-ai/headroom/issues/1495
**评分：** 14/25（深度:3，维护者参与:1，社区关注:3，范围清晰度:3，分析价值:4）

#### 背景

headroom（5.2 万星，2026 年 1 月创建）是一个快速增长的 token 压缩库。其代理层将请求转发到上游 LLM 提供方。当上游返回 HTTP 529（服务过载）时，代理不应用与 HTTP 429（被限速）相同的重试逻辑，导致不必要的失败。

#### 社区评估

- **最近有意义的活动：** 2026-06-27
- **已有实现：** 标题含 "fix:" 前缀——可能是已有的修复 PR，需验证
- **维护者态度：** 未知
- **判断：** 中等。值得关注的快速增长项目。issue 可能已有修复。

#### 问题分析

HTTP 529（服务过载）在语义上与 429（请求过多）相似——两者都表示上游暂时无法处理请求。代理的重试逻辑仅处理 429，因此 529 响应作为失败直接传给客户端，而不是被重试。

#### 修复方案

**根因：** 代理转发器中的重试条件检查状态码 429 但不检查 529。

**修改思路：** 在可重试状态码列表中将 529 添加到 429 旁边。应用相同的指数退避和抖动。

**涉及文件：** 代理转发器模块——重试条件检查

**风险评估：** 极低。一行改动，在条件中添加一个状态码。
