### A1: openclaw/openclaw #43996 — Sandbox container exits immediately with no-new-privileges

**Link:** https://github.com/openclaw/openclaw/issues/43996
**Score:** 18/25 (depth: 4, engagement: 3, interest: 4, clarity: 5, value: 2)

#### Background

OpenClaw (380k stars) is a multi-platform AI agent runtime. Its sandbox feature creates isolated Docker containers for agent code execution. After upgrading to v2026.3.11, sandbox containers fail immediately with `exec /usr/bin/sleep: operation not permitted` (ExitCode=255).

#### Community Assessment

- **Last meaningful activity:** 2026-06-27 (Codex review), user diagnosis contributions dated around issue creation
- **Existing implementation:** None. Multiple users contributed Docker version/environment data for diagnosis
- **Maintainer stance:** Codex bot says "keep open" — no human maintainer engaged yet
- **Verdict:** VIABLE. Rich multi-user diagnosis data available, no one claimed

#### Problem

The sandbox runtime unconditionally adds Docker `--security-opt no-new-privileges` to all containers. On certain host/kernel/Docker combinations, this flag prevents even basic binaries like `/usr/bin/sleep` from executing via `execve`. A user confirmed that removing the flag from their local build resolved the issue immediately.

#### Fix Strategy

**Root cause:** The `--security-opt no-new-privileges` flag is applied unconditionally in the sandbox runtime's Docker invocation code.

**Proposed approach:** Make `no-new-privileges` configurable — either via a sandbox config key (`security.noNewPrivileges: true|false`) or by detecting host compatibility at gateway startup and adjusting the default.

**Files to modify:**
- `src/infra/sandbox/` — Docker container creation, where `--security-opt` flags are assembled
- Configuration schema — add optional `security.noNewPrivileges` toggle

**Risk:** Low. Users on compatible hosts keep existing behavior (secure default). Users on incompatible hosts can disable it. The flag's security benefit is defense-in-depth, not a hard requirement.

### A2: earendil-works/pi #4877 — Session folder collision

**Link:** https://github.com/earendil-works/pi/issues/4877
**Score:** 20/25 (depth: 5, engagement: 5, interest: 3, clarity: 4, value: 3)

#### Background

pi (66k stars) stores session data in folders derived from the working directory path. The path-to-folder encoding replaces `/` with `-`, meaning structurally different paths can produce identical folder names (e.g., `/a/b/c/d` and `/a-b/c-d` both become `--a-b-c-d--`). This causes session data collisions.

#### Community Assessment

- **Last meaningful activity:** Active discussion with MEMBER + 2 COLLABORATORs
- **Existing implementation:** None. Multiple proposals in comments (UUID-based folders, path→UUID mapping table, `pi move` command) but MEMBER commented "No amount of substitution is going to fully resolve that issue"
- **Maintainer stance:** COLLABORATOR said "ugh, thanks for reporting!", MEMBER (mitsuhiko) engaged deeply with the technical difficulty. No "lgtm" or "wont-fix"
- **Verdict:** HIGH VALUE for analysis. Technically challenging problem with active maintainer engagement

#### Problem

The path encoding function conflates `/` and `-` into the same character. A user can have sessions at `/a/b/c/d` and `/a-b/c-d` collide. Additionally, a contributor noted that Windows path length limits make the current encoding approach fragile, and the maintainer noted that any cross-process file locking solution would be "pretty terrible."

#### Fix Strategy

**Three candidate approaches discussed in the thread:**

**Option A — UUID folders + mapping table:** Keep a JSON file mapping `path → uuid` in `~/.pi/`. Sessions stored in UUID-named folders. Solves collisions + Windows long path issue + path rename support. Risk: atomic cross-process updates to the mapping table.

**Option B — `.pi/id` per-project:** Create a UUID at project init, store it in `.pi/id`. Sessions stored under that UUID. No cross-process coordination needed. Risk: requires explicit project initialization.

**Option C — Hash-based encoding:** Replace the `/` → `-` encoding with a hash of the full path (e.g., SHA256 first 12 chars). One-way, no collisions. Risk: loses human-readable folder names.

**Preferred:** Option B is the most "pi-like" — lean, simple, no cross-process locks. Option A is more automatic but introduces coordination complexity.

### A3: HKUDS/nanobot #4307 — Post-turn consolidation wipes agent's delivery message

**Link:** https://github.com/HKUDS/nanobot/issues/4307
**Score:** 16/25 (depth: 4, engagement: 3, interest: 3, clarity: 3, value: 3)

#### Background

nanobot (44k stars) is an AI agent framework by HKUDS. Its context consolidation runs after a turn completes, summarizing the assistant's response into a lossy archive. When the user replies referencing specific options or findings the agent proposed, the agent has no memory of what it said.

#### Community Assessment

- **Last meaningful activity:** COLLABORATOR linked it to broader issue #4044 ("short term memory loss")
- **Existing implementation:** None
- **Maintainer stance:** COLLABORATOR acknowledged as a "concrete subcase" of the broader memory loss issue
- **Verdict:** VIABLE. Real UX pain point with specific diagnosis

#### Problem

When `context_window_tokens` is modest (e.g., 40k), a long multi-iteration turn can accumulate 100k+ tokens before consolidation runs. Consolidation fires only after turn completion and archives the assistant's delivery message (options, findings, proposals) into a lossy summary. On the next turn, the user references those proposals and the agent falls back to stale or incorrect context.

#### Fix Strategy

**Root cause:** The consolidation trigger is token-count-based but fires too late — after the turn, not during it. Additionally, the consolidation prompt asks the agent to extract "key points," which loses detail.

**Proposed approach:**
1. Add a mid-turn consolidation checkpoint that fires at a lower threshold (e.g., 70% of context window) instead of only post-turn
2. Preserve the last N assistant messages in full (not summarized) to maintain conversational continuity
3. Adjust the consolidation prompt to retain action items and proposals explicitly

**Files to modify:**
- Consolidation trigger logic (likely in a `consolidator` or `memory` module)
- Consolidation prompt template (`consolidator_archive.md` mentioned in comments)

**Risk:** Medium. Changing consolidation behavior affects all users. Needs configurable thresholds.

### B1: fastfetch-cli/fastfetch #2423 — GPU detection wakes dedicated GPU on dual-GPU laptops

**Link:** https://github.com/fastfetch-cli/fastfetch/issues/2423
**Score:** 14/25 (depth: 3, engagement: 1, interest: 4, clarity: 3, value: 3)

#### Background

fastfetch (15k stars) is a popular system information tool (neofetch successor). On laptops with dual GPUs (integrated + dedicated NVIDIA), fastfetch pings the dedicated GPU during information gathering, causing the dGPU to wake from power-saving state. This makes fastfetch perceptibly slow and wastes battery.

#### Community Assessment

- **Last meaningful activity:** Recent (June 2026)
- **Existing implementation:** None
- **Maintainer stance:** No maintainer response yet (1 comment from reporter)
- **Verdict:** VIABLE. Clear system-level bug, well-defined repro conditions

#### Problem

After updating to kernel 7.1.1 + latest NVIDIA open driver, fastfetch's GPU detection path activates the dedicated GPU even when it's in a suspended/power-saving state. This didn't happen before the kernel/driver update, suggesting a change in how the NVIDIA driver exposes GPU state that fastfetch's detection code doesn't account for.

#### Fix Strategy

**Root cause:** fastfetch's GPU detection module likely reads GPU properties via sysfs or NVML, which triggers a wake on the NVIDIA side with the latest open driver.

**Proposed approach:**
1. Check GPU power state before querying detailed properties — if suspended, skip detailed queries and report "suspended" or use cached data
2. Use `nvidia-smi --query-gpu=power_state` or sysfs `/sys/class/drm/card*/device/power_state` to check state first
3. Optionally add a `--gpu-detection=light` flag that skips wake-triggering queries

**Files to modify:**
- GPU detection module (likely `src/detection/gpu/` or similar)
- Possibly `src/modules/gpu/` for the display module
