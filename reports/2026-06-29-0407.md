### A1：earendil-works/pi #6083 — z.ai GLM Coding Plan 的 LLM 缓存未正常工作

**链接：** https://github.com/earendil-works/pi/issues/6083
**评分：** 18/25（深度:4，维护者参与:2，社区关注:5，范围清晰度:3，分析价值:4）

#### 背景

pi（6.6 万星）在使用 z.ai GLM Coding Plan 时，每次工具调用都会大量消耗 session 配额。纯对话轮次还算正常，但涉及多次 write/edit 工具调用的多步骤任务，即使活跃上下文低于 50k，也很容易每次任务消耗 10-20% 的 session 配额。在 OpenClaw 中使用同样的 z.ai 工作流程并无此问题，说明这是 pi 特有的缓存失效问题。

#### 社区评估

- **最近有意义的活动：** 有社区成员提出了补丁，并自行测试确认"配额使用量大幅降低——正在恢复到配额限制内"
- **已有实现：** 社区补丁存在且经实测有效，但未被合并
- **维护者态度：** issue 被 contributor gate 自动关闭，等待 maintainer 审查
- **判断：** 高价值。👍10 反应（本次扫描最高），已有社区补丁验证有效

#### 问题分析

z.ai GLM Coding Plan 的 API 与 pi 的 LLM 缓存机制存在兼容性问题。具体表现为工具调用场景下缓存命中率异常低，导致每次工具轮次都重新消耗完整上下文 token。纯对话轮次受影响较小，说明问题集中在工具调用相关的 prompt 构造环节——可能是工具定义、工具结果或系统提示的缓存键不稳定。

#### 修复方案

**根因：** 工具调用场景下的 prompt 缓存键可能包含了动态变化的内容（如 timestamp、递增计数器、或每次工具调用都变化的上下文摘要），导致缓存持续失效。

**修改思路：**
1. 对比 pi 和 OpenClaw 在 z.ai API 上的缓存行为差异（请求头、prompt 结构）
2. 检查工具调用轮次中 prompt 缓存的 cache breakpoint 位置是否合理
3. 社区补丁已验证有效——可作为修复起点，maintainer 需审查并合并
4. 添加针对 z.ai provider 的缓存命中率监控/日志

**涉及文件：** z.ai provider 的缓存配置、工具调用的 prompt 构造模块

**风险评估：** 低。社区已实测补丁有效，不会引入功能回归。

### A2：openclaw/openclaw #77642 — lossless-claw 回归：重复回答 + "missing tool result" 合成错误

**链接：** https://github.com/openclaw/openclaw/issues/77642
**评分：** 19/25（深度:5，维护者参与:2，社区关注:4，范围清晰度:4，分析价值:4）

#### 背景

OpenClaw（38 万星）5.2 版本引入的回归（持续到 5.3.1）。两个相关症状：(1) 用户追问时，回答中包含上一次回答的重复内容；(2) 出现 "No result provided" / "missing tool result in session history" 合成错误。lossless-claw 功能（lossless context preservation）与此强相关——Codex 审查确认当前 main 分支的 `transport-message-transform.ts` 中仍有 `syntheticToolResultText` 生成器。

#### 社区评估

- **最近有意义的活动：** 2026-06-28，多位用户在 Telegram 等多个渠道确认
- **已有实现：** Codex 审查指出 `src/agents/transport-message-transform.ts` 中存在 `"No result provided"` 生成器
- **维护者态度：** Codex bot "保持开放"，无人类维护者
- **判断：** 高价值。用户原话 "agent becomes so dumb that I rolled back"——影响严重到用户回退版本

#### 问题分析

两个症状共享同一根因——lossless-claw 的消息转换管线在处理工具结果时产生了错误的合成消息。当真实工具结果丢失或延迟到达时，`transport-message-transform.ts` 中的 `syntheticToolResultText` 生成器插入 "No result provided" 占位文本。但该占位文本随后被当作真实工具结果处理，导致：(1) 上下文污染，模型基于错误历史生成重复回答；(2) 后续工具调用引用不存在的工具结果时触发 "missing tool result" 错误。

#### 修复方案

**修改思路：**
1. 在 `transport-message-transform.ts` 中定位 `syntheticToolResultText` 生成逻辑
2. 区分"工具结果确实为空"和"工具结果尚未到达"两种情况——前者是合法的（某些工具无返回值），后者应等待而非合成
3. 对合成消息添加标记，防止被后续管线误当作真实历史
4. 添加集成测试覆盖工具结果丢失/延迟到达的场景

**涉及文件：** `src/agents/transport-message-transform.ts`，lossless-claw 的消息恢复管线

**风险评估：** 中等。涉及核心消息管线，修改需谨慎以免引入新的上下文损坏。但有明确的代码定位和多位用户的复现确认。

### A3：openclaw/openclaw #77733 — `/new` 和 `/reset` 不再触发 persona 问候语（5.3 回归）

**链接：** https://github.com/openclaw/openclaw/issues/77733
**评分：** 16/25（深度:3，维护者参与:2，社区关注:4，范围清晰度:4，分析价值:3）

#### 背景

OpenClaw 2026.5.3 版本中，执行无参数的 `/new` 或 `/reset` 命令不再触发 agent 的 persona 问候语。用户收到的是预设文本回复 `✅ New session started.` / `✅ Session reset.`，且**没有 LLM 轮次运行**。而在 4.x 版本中，同样的命令会触发 persona 系统提示，让 agent 以个性化问候开始新会话。

#### 社区评估

- **最近有意义的活动：** 2026-06-28
- **已有实现：** 无
- **维护者态度：** Codex bot "保持开放"
- **判断：** 可行。清晰的功能回归，影响用户体验。技术范围小，适合分析。

#### 问题分析

带参数 `/new <指令>` 时工作正常（agent 执行指令），但裸 `/new` 和 `/reset` 被短路处理——系统直接返回预设文本而不调用 LLM。问题出在命令路由层：裸 `/new` 被当作"不需要 LLM 的元命令"处理，而 4.x 中它被路由为"用 persona 提示词发起新的 agent 轮次"。

#### 修复方案

**修改思路：**
1. 在命令路由中定位 `/new` 和 `/reset` 的处理逻辑
2. 裸命令（无参数）应触发 persona 问候流程：创建新 session → 注入 persona 系统提示 → 发起 LLM 轮次
3. 带参数命令保持现有行为（直接执行指令）
4. 可选：添加配置项控制裸 `/new` 是否触发问候，让用户选择偏好

**涉及文件：** 命令路由/处理器（可能在 `src/commands/` 或频道适配器层）

**风险评估：** 低。修改范围小，仅影响两个命令的无参数情况。

#### 背景

socktainer 是一个 Docker 套接字代理工具，用于在容器间安全共享 Docker 套接字。其文档描述了一个 Docker Compose 内部服务 DNS 解析的示例，但多位用户报告该示例无法工作——容器间无法通过服务名互相解析。

#### 社区评估

- **维护者态度：** 未知

#### 问题分析

Docker Compose 的 DNS 解析依赖自定义网络。如果 socktainer 代理改变了网络配置（例如将容器附加到不同的网络），则 Compose 内置的 DNS（`<服务名>`）将无法解析。问题可能出在代理启动时的网络配置步骤。

### B2：elastic/kibana #274925 — 安全规则变更历史缺少基线快照

**链接：** https://github.com/elastic/kibana/issues/274925
**评分：** 16/25（深度:4，维护者参与:1，社区关注:3，范围清晰度:5，分析价值:3）

#### 背景

Elastic Kibana（弹性搜索官方项目）的安全解决方案中，规则变更历史功能通过在每次 CRUD 操作时捕获完整规则快照来追踪变化。然而，功能启用前就已存在的所有规则都没有基线快照——它们的变更历史为空。这意味着无法查看"功能启用前规则是什么样的"，也无法计算首次变更的差异。

#### 社区评估

- **最近有意义的活动：** 近期
- **已有实现：** 无
- **维护者态度：** 未知
- **判断：** 可行。清晰的工程问题，Elastic 是大项目，有良好的 issue 规范。

#### 问题分析

变更历史系统使用事件溯源模式——每个 CRUD 操作写入一个快照。读取时通过比较连续快照（RFC 7396 JSON Merge Patch）计算差异。问题在于系统只在 CRUD 操作时写入快照，而功能启用本身不是 CRUD 操作——所以已有规则从未获得初始快照。

#### 修复方案

**修改思路：**
1. 在功能首次启用时（或升级时），为所有现有规则批量生成初始基线快照
2. 这可以作为一次性迁移任务运行
3. 业务逻辑：对于没有基线快照的规则，将其当前状态作为"基线"，后续变更才能比较

**涉及文件：** 规则变更历史模块、迁移/升级脚本

**风险评估：** 低。一次性数据迁移，不影响运行时行为。

### B3：NousResearch/hermes-agent #54354 — Docker 后端：镜像拉取前的首次工具调用在宿主机上运行

**链接：** https://github.com/NousResearch/hermes-agent/issues/54354
**评分：** 17/25（深度:5，维护者参与:1，社区关注:3，范围清晰度:4，分析价值:4）

#### 背景

Hermes Agent（20.5 万星）的 Docker 后端采用懒加载方式创建沙箱容器——在首次工具调用时才拉取镜像并创建容器。严重问题：在镜像拉取完成之前，首次工具调用**直接在宿主机上执行**，返回的是宿主机路径（如 `~/username/...`）而非容器内路径。第二次调用时镜像已就绪，才正常在容器内执行。

#### 社区评估

- **最近有意义的活动：** 2026-06-28（新 issue）
- **已有实现：** 无
- **维护者态度：** 无回复
- **判断：** 高价值。**实质上是沙箱逃逸**——在沙箱就绪前，代码在宿主机上运行。安全隐患。

#### 问题分析

Docker 沙箱的懒加载逻辑存在竞态条件：(1) 用户发送需要工具调用的消息；(2) 系统开始拉取 Docker 镜像（异步）；(3) 在镜像拉取完成之前，工具调用请求已到达执行层；(4) 执行层发现容器不存在，回退到宿主机直接执行。正确的行为应该是：在镜像拉取完成之前**阻塞**工具调用，或在容器未就绪时返回明确错误而非回退到宿主机。

#### 修复方案

**修改思路：**
1. 在工具执行入口添加"沙箱就绪"守卫——如果 Docker 后端配置了但容器未运行，阻塞等待或返回错误
2. 将镜像拉取从懒加载改为启动时预加载（gateway 启动时拉取镜像）
3. 添加明确的宿主机回退禁用开关——当 Docker 后端配置了但不可用时，必须失败而非回退

**涉及文件：** Docker 沙箱后端模块、工具执行入口

**风险评估：** 低（从安全角度——这是修漏洞）。实现上需注意不要引入过长的启动等待时间。
