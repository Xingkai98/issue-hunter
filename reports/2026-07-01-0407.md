### A1：openclaw/openclaw #38327 — Google Vertex 回归："Cannot convert undefined or null to object"

**链接：** https://github.com/openclaw/openclaw/issues/38327
**评分：** 18/25（深度:4，维护者参与:3，社区关注:4，范围清晰度:4，分析价值:3）

#### 背景

升级到 2026.3.2 后，使用 google-vertex/gemini-3.1-pro-preview 模型的 embedded agent 每次消息都崩溃，报 `"Cannot convert undefined or null to object"`。👍3，9 条评论，3 月报告至今仍然活跃。标题含 `[Bug]` 但未标 label:bug（说明 issue 创建于旧模板时代）。

#### 社区评估

- **活跃度：** 👍3，9 条评论，仍在活跃讨论
- **已有实现：** 无
- **维护者态度：** 无人类维护者
- **判断：** 高价值。影响 google-vertex 用户核心体验的长期回归

#### 问题分析

错误发生在 Google Vertex AI SDK 与 OpenClaw 集成层的某个地方——将 `undefined` 或 `null` 传给期望对象的函数。可能的位置：
1. Vertex AI 响应解析——模型返回了不符合预期的响应结构，某个嵌套字段缺失
2. 消息转换层——`transformMessages` 在将消息转为 Vertex 格式时，某个必要字段未填充
3. 与 #38327 相关的 3.2 版本变更——对比 3.1 到 3.2 的 diff 中 google-vertex 相关文件的变更

#### 修复方案

**根因排查：** 对比 2026.3.1 和 2026.3.2 之间 google-vertex 相关的 diff，重点查消息转换和响应解析逻辑。

**修改思路：**
1. 在崩溃点添加防守代码——对可能为 `undefined` 的对象使用可选链（`?.`）和空值合并（`??`）
2. 添加日志输出实际收到的 Vertex 响应结构，帮助定位哪个字段缺失

**涉及文件：** google-vertex provider 适配器

**风险评估：** 低。防御性编程，不改变业务逻辑。

### A2：NousResearch/hermes-agent #55677 — 上下文压缩 Jinja 模板错误导致会话损坏

**链接：** https://github.com/NousResearch/hermes-agent/issues/55677
**评分：** 16/25（深度:4，社区关注:2，范围清晰度:4，分析价值:4，维护者参与:2）

#### 背景

Hermes Agent 的内部上下文压缩在第 2-3 次压缩时崩溃，报 `"No user query found in messages"` Jinja 模板错误。会话被损坏，对话连续性丢失，必须新建会话。0 评论，非常新的 bug。

#### 问题分析

上下文压缩（context compaction）将长对话历史总结为更短的摘要以节省 token。压缩过程使用 Jinja 模板渲染 prompt。第 1 次压缩成功，第 2-3 次失败——说明第 1 次压缩后的输出格式与模板预期不匹配。模板期望在消息列表中找到 `user query` 字段，但压缩后的历史摘要中该字段缺失或结构不同。

#### 修复方案

**修改思路：**
1. 检查压缩模板中的 `user query` 引用——该字段是否在压缩后的消息结构中仍存在
2. 在压缩输出时确保保留 `user query` 字段（即使为空字符串）
3. 或在模板中添加 `user query` 的 fallback 逻辑

**涉及文件：** 上下文压缩模块，Jinja 模板

**风险评估：** 低。修复范围限定在压缩模板。

### A3：earendil-works/pi #6187 — WSL 环境下 GitHub Copilot 设备授权后登录挂起

**链接：** https://github.com/earendil-works/pi/issues/6187
**评分：** 14/25（深度:3，社区关注:2，范围清晰度:3，分析价值:3，维护者参与:3）

#### 背景

在 WSL 中运行 pi 时，浏览器端 GitHub Copilot 设备授权成功完成（设备显示为已注册），但 WSL 终端中的 pi 客户端未能检测到授权完成，一直挂起等待登录完成。1 条评论，新鲜 issue。

#### 问题分析

WSL 环境与 Windows 宿主之间的网络边界可能导致回调检测失败。pi 的设备授权流程可能使用本地 HTTP 回调或轮询机制来检测授权完成——在 WSL 中，localhost 端口映射或回调 URL 可能无法从 Windows 浏览器正确路由回 WSL 进程。

#### 修复方案

**修改思路：**
1. 检查 pi 的 OAuth 回调机制——是使用本地 HTTP 服务器还是轮询
2. WSL 中可能需要使用 `wslhost` 或显式 IP 绑定而非 localhost
3. 添加超时和轮询重试机制，而非无限等待

**涉及文件：** Copilot 认证/OAuth 模块

**风险评估：** 低。平台特定修复。
