### A1：openclaw/openclaw #98540 — Composer 在 agent 执行工具时显示空闲状态

**链接：** https://github.com/openclaw/openclaw/issues/98540
**评分：** 16/25（深度:3，社区关注:3，范围清晰度:4，分析价值:3，维护者参与:3）

#### 背景

当 agent 正在执行工具（Activity 面板显示 "N tools Exec"）时，Composer 区域不显示 "In progress" 指示器，也看不到正在运行的工具。Composer 表现为空闲状态，但实际 agent 正在活跃工作。UI 状态同步 bug。

#### 问题分析

Composer 的状态机可能使用不同的信号源来判断 agent 是否忙碌——它可能监听 LLM 流式输出事件而非工具执行事件。当 agent 进入工具调用阶段（LLM 停止流式输出），Composer 认为 agent 空闲了，但实际上工具正在后台运行。

#### 修复方案

在 Composer 的状态判断中添加工具执行状态作为忙碌信号源。检查 `Activity` 面板和 `Composer` 组件是否使用了统一的 agent 状态源。

### A2：NousResearch/hermes-agent #56516 — 流式推理模型错误触发"Provider returned an empty stream"

**链接：** https://github.com/NousResearch/hermes-agent/issues/56516
**评分：** 15/25（深度:4，社区关注:2，范围清晰度:4，分析价值:4，维护者参与:2）

#### 背景

使用推理模型（reasoning models）时，Hermes Desktop 错误抛出 `RuntimeError: Provider returned an empty stream with no finish_reason`。这些模型在发出 `content` 之前只流式输出 `reasoning_content`（思考链），导致 Hermes 的流处理器在看到任何文本内容之前就判断流为空。

#### 问题分析

推理模型的流式响应有两阶段結構：先 `reasoning_content` 再 `content`。Hermes 的流处理逻辑可能在收到第一个 chunk 时检查是否有内容——如果第一个（或前几个）chunk 只包含 `reasoning_content` 而 `content` 为空，就错误地判为"空流"。正确的行为应该是：持续等待直到看到 `content` 或 `finish_reason`。

#### 修复方案

在流处理循环中区分"尚未收到内容"和"流确实为空"——收到 `reasoning_content` 不应触发空流判断，应继续等待直到 `content` 非空或 `finish_reason` 到达。

### A3：earendil-works/pi #6206 — 上下文窗口限制与 maxTokens 的交互问题

**链接：** https://github.com/earendil-works/pi/issues/6206
**评分：** 15/25（深度:4，社区关注:3，范围清晰度:3，分析价值:4，维护者参与:2）

#### 背景

pi 的上下文限制修复（commit `09f1059`）将 `maxTokens` 压缩到上下文窗口限制，但这可能导致人为的上下文限制——高配模型的上下文窗口远超实际需要时，压缩后的 `maxTokens` 仍然远低于模型的真实能力。问题在于 `maxTokens` 和上下文窗口是两个不同的概念，但修复将它们耦合在一起。

#### 问题分析

`maxTokens` 控制模型输出的最大 token 数，上下文窗口控制输入+输出的总 token 数。将 maxTokens 压缩到上下文窗口减去当前上下文的剩余空间是正确的——但当模型有巨大的上下文窗口（如 200K）而实际上下文只有 20K 时，压缩后的 maxTokens 可能仍然低于模型支持的输出上限。修复方式应该是：使用模型的输出 token 上限（如 16K）作为 maxTokens 的上限，而非上下文窗口。

#### 修复方案

使用 `min(模型支持的maxTokens, 上下文窗口 - 当前上下文)` 而非单纯的上下文窗口限制。

### B2：directus/directus #27804 — 几何类型在更改"允许 NULL"时被重置

**链接：** https://github.com/directus/directus/issues/27804
**评分：** 13/25（深度:3，社区关注:2，范围清晰度:4，分析价值:3，维护者参与:2）

#### 背景

Directus 是一个流行的开源 CMS/数据平台。在 collections 编辑器中，更改几何类型字段的 "Allow NULL" 选项时，字段类型被重置为默认类型（ANY），丢失了原有的几何类型配置（如 Point、Polygon 等）。

#### 问题分析

前端表单在提交 "Allow NULL" 变更时，未保留原有的字段类型值——可能整个字段配置对象在 PATCH 时只发送了修改的字段，但后端在合并时使用了错误的默认值。

#### 修复方案

检查字段编辑 API 的 PATCH 请求体和后端合并逻辑——确保部分更新时保留未修改的字段类型。

### B3：bitfocus/companion #3398 — 模拟器按钮被多次触发

**链接：** https://github.com/bitfocus/companion/issues/3398
**评分：** 12/25（深度:3，社区关注:3，范围清晰度:3，分析价值:3，维护者参与:2）

#### 背景

Companion 是一个直播/演播控制软件。Emulator（模拟器）中的按钮点击事件被重复触发多次，而非单次触发。7 条评论，多位用户确认。

#### 问题分析

事件监听器可能未正确 debounce，或组件在重新渲染时注册了重复的事件处理器。可能是 React 组件生命周期问题——每次渲染时为按钮注册新的事件处理器，旧处理器未被清理。

#### 修复方案

检查 Emulator 按钮组件的事件注册逻辑，确保在组件卸载时移除事件监听器（useEffect cleanup），或使用事件委托避免多次注册。
