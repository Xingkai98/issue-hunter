## Issue 1: openclaw/openclaw #40255 — Regression: User-configured heartbeat prompt no longer respected

**Link:** https://github.com/openclaw/openclaw/issues/40255
**Score:** 17/20 (depth: 4, engagement: 3, interest: 4, clarity: 4, value: 2)

### Background

OpenClaw (380k stars) is a multi-platform AI agent runtime written in TypeScript. Its "heartbeat" system periodically polls agents for status. Users can configure a custom heartbeat prompt (e.g., to reference a `HEARTBEAT.md` file from the system prompt instead of the workspace). Commit `604f22c` introduced a feature that unconditionally appends a file-path hint to the heartbeat prompt:

> "When reading HEARTBEAT.md, use workspace file ${heartbeatFilePath} (exact case). Do not read docs/heartbeat.md"

This *overrides* the user's custom heartbeat prompt configuration, breaking workflows that depend on custom heartbeat behavior.

### Problem

The affected file is `src/infra/heartbeat-runner.ts`. The fix in commit `604f22c` added the workspace-path hint text unconditionally — it does not check whether the user has already configured a custom heartbeat prompt. If a user has set a custom prompt (e.g., instructing the agent to read HEARTBEAT.md from a different location or with different instructions), the appended text contradicts or corrupts their configuration.

**Repro steps (from issue):**
1. Set a custom heartbeat prompt
2. Observe the text "When reading HEARTBEAT.md, use workspace file ${heartbeatFilePath}..." being added to it unconditionally
3. The agent follows the appended text instead of the user's custom instructions

**Maintainer engagement:** Codex AI triage bot reviewed it and flagged it as "keep open." One community member cross-verified the regression across multiple versions (v2026.6.6, v2026.6.8-beta.1). No human maintainer has commented yet, but the issue has 5 comments and 👍2 reactions.

### Fix Strategy

**Root cause:** The heartbeat prompt assembly in `src/infra/heartbeat-runner.ts` does not check for an existing user configuration before appending the path hint.

**Proposed approach:** Add a conditional gate — if the user has configured a custom heartbeat prompt (via `resolveHeartbeatPrompt()`), skip the automatic workspace-path hint. Only append the hint when the user has NOT set a custom prompt.

**Files to modify:**
- `src/infra/heartbeat-runner.ts` — add the conditional gate before appending the path hint text
- `src/infra/heartbeat-runner.returns-default-unset.test.ts` — existing test file, add a test case for the custom-prompt-unchanged behavior

**Risk assessment:** Low. The fix is a simple guard clause. The regression was introduced in a single commit that modified only 3 files. Rolling back the unconditional append or gating it behind a config check restores the pre-regression behavior without affecting the default path (users who rely on the automatic hint still get it, users with custom prompts don't get overridden).

## Issue 2: earendil-works/pi #5825 — Streaming markdown forces scroll to bottom

**Link:** https://github.com/earendil-works/pi/issues/5825
**Score:** 18/20 (depth: 5, engagement: 5, interest: 4, clarity: 4, value: 0)

### Background

pi (66k stars) is an extensible AI agent toolkit by Earendil, written in TypeScript. Its TUI (Terminal User Interface) renders streaming markdown output from the agent. When the `clearOnShrink` setting is enabled, incomplete markdown elements (e.g., code fences with only the opening backticks but not the closing ones) cause the rendered output to "shrink" — previously wrapped lines collapse to fewer lines, triggering a full re-render that forces a scroll-to-bottom.

### Problem

**Affected subsystem:** `packages/tui/src/tui.ts` — the `Container.doRender` method.

**Root cause chain:**
1. During streaming markdown, incomplete code fences (e.g., `` ``` `` without a closing `` ``` ``) cause the markdown parser to produce token trees that include large "paragraph" blocks
2. When the closing fence arrives, the token tree shrinks — a multi-line paragraph collapses into a single-line code block
3. `clearOnShrink` detects the line count decrease and calls `fullRender(true)` which resets the scroll position to the bottom
4. The user, who was reading up above, is yanked back to the bottom

**Maintainer engagement:** HIGH. Two MEMBERs (including a maintainer) confirmed reproduction and engaged deeply. One maintainer started a WIP fix branch (`fix/5825-tui-shrink-redraw`). A community contributor (@vegarsti) opened PR #5846 with a self-contained solution: walk the markdown token tree, trim trailing incomplete code fences before rendering. The maintainer acknowledged the approach is promising but the PR is still in discussion.

The discussion has 33 comments — one of the most actively discussed issues in pi.

### Fix Strategy

**Option A (PR #5846 approach — preferred):** Trim trailing incomplete code fences from the markdown token tree before rendering. This prevents the "shrink" event from happening in the first place.

- Walk the parsed markdown token tree
- For each code block, check if the trailing fence is incomplete (opening backticks but no closing)
- Trim the incomplete part before passing to the renderer

**Option B (TUI-level fix):** Modify `fullRender(true)` to preserve the user's scroll position when the content shrinks, rather than forcing a scroll-to-bottom. This is the approach on the WIP branch `fix/5825-tui-shrink-redraw` but the maintainer described it as "still slop."

**Files to modify:**
- `packages/tui/src/tui.ts` — `Container.doRender`, the `clearOnShrink` logic
- Possibly `packages/tui/src/markdown-renderer.ts` — if going with Option A

**Risk assessment:** Medium. The markdown rendering path is complex and markdown parsers have edge cases. Option A (trim incomplete fences) is more surgical than Option B (change scroll behavior) and has an existing PR to reference. The main risk is that other markdown elements (long URLs, inline images) can also shrink on re-render, which the PR #5846 discussion acknowledges.

## Issue 3: karpathy/nanochat #328 — Fix relative difference sign in scripts/tok_eval.py

**Link:** https://github.com/karpathy/nanochat/issues/328
**Score:** 15/20 (depth: 3, engagement: 3, interest: 3, clarity: 5, value: 1)

### Background

nanochat (55k stars) is Andrej Karpathy's minimal LLM implementation in Python, designed for learning and understanding LLM internals. `scripts/tok_eval.py` is a tokenizer evaluation script that compares compression ratios across tokenizers (GPT-2, GPT-4, and the project's custom tokenizer).

### Problem

In `scripts/tok_eval.py`, the relative difference calculation has its numerator sign flipped:

```python
# Current (buggy):
relative_diff = ((baseline_data['tokens'] - ours_data['tokens']) / baseline_data['tokens']) * 100
```

When "ours" uses 3X the tokens of the baseline (worse compression), this reports `-200%`, which reads as "ours is 200% better" when it should read "ours is 200% worse."

The fix:
```python
relative_diff = ((ours_data['tokens'] - baseline_data['tokens']) / baseline_data['tokens']) * 100
```

Now positive means ours uses more tokens (worse), negative means ours uses fewer (better). This aligns with the natural intuition of "relative difference from baseline."

**Maintainer engagement:** A COLLABORATOR initially pushed back but after detailed back-and-forth with the reporter, agreed: "I had another detailed look into this, and I think you're right. I got confused by the sign switch."

### Fix Strategy

**Single-line change** in `scripts/tok_eval.py` line 217:

```python
# Before:
relative_diff = ((baseline_data['tokens'] - ours_data['tokens']) / baseline_data['tokens']) * 100

# After:
relative_diff = ((ours_data['tokens'] - baseline_data['tokens']) / baseline_data['tokens']) * 100
```

**Additionally:** The color logic on lines 219-227 needs to be inverted to match — currently `baseline_data['ratio'] > ours_data['ratio']` sets green for baseline and red for ours (correct), but after the sign flip, the `better` label logic should be based on the sign of `relative_diff` rather than comparing ratios separately.

**Files to modify:**
- `scripts/tok_eval.py` — line 217 (formula) + lines 219-227 (color logic)

**Risk assessment:** Trivial. One-line math fix with corresponding presentation logic adjustment. The COLLABORATOR has already agreed the fix is correct. Run `uv run -m scripts.tok_eval` to verify the output signs make sense.
